# ADR 0010: 목록 조회 전략 — 커서 페이지네이션 (QueryDSL 배제)

## 상태 (Status)

Accepted (2026-07-16)

## 배경

무한 스크롤 목록이 여럿 필요하다(현위치 장소, 저장한 장소, 추후 리뷰 등). 실시간 삽입에도 페이지 경계가 흔들리지 않아야 하고, 공간 목록은 거리·범위를 DB 인덱스로 처리해야 한다. 쿼리 라이브러리 방침도 정해야 한다.

## 결정

- **무한 스크롤 목록은 커서(keyset) 페이지네이션을 기본**으로 한다. offset은 쓰지 않는다.
  - 정렬 키를 기준으로 다음 페이지를 잇고, 커서는 불투명 토큰으로 주고받는다(응답 `nextCursor`, null=마지막).
  - **정렬 값이 같으면 `id` 오름차순(작은 id 먼저)** 을 타이브레이커로 두어 순서를 결정적으로 만든다.
- **페이지 크기 `size`는 요청 파라미터**로 받는다(기본값 있음).
- **총 개수 `totalCount`는 필요한 목록에만** 응답에 포함한다(예: 지도 클러스터 배지 대조가 필요한 장소 목록). 북마크·리뷰 등은 생략 가능. 포함하는 경우에도 **첫 페이지에서만 계산**하고 이후 페이지는 생략한다(매 페이지 count 쿼리 방지).
- **공간 목록(장소)은 PostGIS**로 뷰포트 범위 필터와 현위치 거리순 정렬을 DB 인덱스로 처리한다([ADR 0003](0003-region-hierarchy-and-postgis.md)).
- **QueryDSL은 도입하지 않는다.** Spring Data 기본에 필요 시 native SQL·Specifications로 충분하다.

## 적용 범위

무한 스크롤로 이어 받는 목록은 모두 커서를 쓰며 정렬 키만 다르다.

| 목록 | 정렬 키 |
|------|---------|
| 현위치 장소 목록 | 거리 (+ id) |
| 저장한 장소(북마크) 목록 | 저장 시각 (+ id), 최신순 |
| 리뷰 목록(추후) | 작성 시각 (+ id), 최신순 |

전체를 한 번에 주는 소규모 고정 목록(예: 지도 마커 전량 반환)은 커서를 쓰지 않는다.

## 결과

### 긍정적

- 삽입/삭제 중에도 페이지 경계가 안정적(offset의 중복·누락·깊은 페이지 성능저하 없음).
- 공간 연산을 DB 인덱스로 처리 → 앱 메모리 정렬 불필요.
- QueryDSL 미도입으로 의존성·빌드가 단순.

### 부정적

- 커서는 임의 페이지 점프 불가(무한 스크롤엔 무방).
- 결정적 정렬 키가 없으면 쓸 수 없어 `id` 타이브레이커가 필수.

### 중립적

- 정렬 키가 도메인마다 달라도 커서 패턴은 동일하게 재사용.

## 검토한 대안

### 대안 1 — Offset 페이지네이션

단순하지만 삽입 시 중복/누락, 깊은 페이지 성능저하로 무한 스크롤에 부적합.

### 대안 2 — QueryDSL

타입 안전·동적 쿼리에 강하나 공간 함수는 우회가 필요하고 Q타입 생성·의존성 부담이 있어 미도입.
