# ADR 0008: 소셜 로그인 전략

## 상태 (Status)

Accepted (2026-07-03)

## 배경

소셜 로그인 기반 모바일 서비스다. 클라이언트가 소셜 공급자(카카오 우선, 애플 등) 토큰을 서버에 전달하면, 서버는 이를 검증해 내부 회원을 식별한다. 다음을 결정해야 한다.

- 여러 공급자를 어떻게 **확장 가능하게** 지원할 것인가 (새 공급자 추가 비용)
- 한 사용자가 **여러 공급자 계정을 연결**할 수 있게 할 것인가, 그렇다면 데이터를 어떻게 나눌 것인가
- 회원을 식별하는 **안정적 기준**은 무엇인가

로그인 이후의 서비스 인증(자체 토큰)은 [ADR 0009](0009-authentication-authorization.md)에서 다룬다. 본 문서는 **공급자로 신원을 확인해 내부 회원을 식별하는 전략**에 대한 결정이다.

## 결정

**1. 소셜 공급자 검증으로 내부 회원을 식별하고, 서비스 인증과 분리한다.**
공급자 토큰은 **로그인 시 회원을 식별**하는 데만 쓴다. 식별 후에는 서버 자체 토큰을 발급([ADR 0009](0009-authentication-authorization.md))하며, 이후 서비스 API는 공급자 토큰을 다시 요구하지 않는다.

**2. 공급자별 검증을 공통 추상화(전략) 뒤에 두어 확장을 국소화한다.**
공급자마다 검증 방식이 다르므로(카카오와 애플의 토큰 검증 절차가 상이), 공급자별 로직을 **공통 인터페이스 뒤로 분리**한다. 새 공급자 추가는 해당 전략 구현을 더하는 **국소 변경**으로 끝나고, 로그인 오케스트레이션 등 코어 흐름은 바뀌지 않는다.

**3. 소셜 계정과 회원을 분리한다 (1:N).**
회원(내부 신원·프로필)과 소셜 계정(공급자 + 공급자 사용자 ID)을 **별도 개념으로 분리**해, 한 회원이 여러 공급자 계정을 연결할 수 있게 한다. 회원에 공급자 정보를 직접 박아두면 다중 연결이 불가능하므로 택하지 않는다. (연결 기능 자체의 구현은 후속 범위)

**4. 안정 식별자는 공급자의 사용자 ID다 — 이메일은 신뢰 기준으로 쓰지 않는다.**
회원 매칭 기준은 **공급자가 부여한 사용자 ID**로 삼는다. 이메일은 공급자·동의 상태에 따라 없거나 바뀔 수 있어(애플은 최초 1회만 제공, 카카오는 동의 필요) 식별 기준으로 부적합하며, 부가 프로필로만 취급한다.

> 공급자별 검증 절차, 키 관리, 신규/기존 회원 분기 처리 등 구현 상세는 본 ADR 범위 밖이며 구현 단계에서 정한다.

## 결과

### 긍정적
- 새 공급자 추가가 국소적(전략 구현 추가)이라 확장 비용이 낮다.
- 회원↔소셜 계정 분리로 **다중 연결**을 자연스럽게 지원한다.
- 공급자 사용자 ID 기준 식별이라 이메일 변동·미제공에 영향받지 않는다.

### 부정적
- 공급자마다 검증 로직을 유지보수해야 한다(공급자 정책·키 변경 추적 필요).
- 회원·소셜 계정 분리로 조회·정합성 관리가 단일 테이블보다 복잡하다.

### 중립적
- 계정 연결·탈퇴 등 파생 기능은 후속 범위로 남긴다.
- 카카오를 먼저 도입하고, 애플은 동일 추상화 위에 후속으로 추가한다.

## 갱신 (2026-07-05) — 애플 추가·프로필·탈퇴 결정

카카오 구현·운영 배포 후, 애플 도입과 회원탈퇴를 준비하며 다음을 결정했다(상세는 [애플 로그인 스펙](../specs/002-apple-login.md)·[회원 탈퇴 스펙](../specs/003-member-withdrawal.md)).

**5. 공급자별 검증 방식은 목적에 맞게 다르게 둔다(전략 뒤에서).**
카카오는 access token 검증(`/v2/user/me`), **애플은 `authorizationCode` 교환**을 택한다. 애플을 code 교환으로 하는 이유는 **탈퇴 시 애플 토큰 revoke가 필수**(App Store 심사)이고 그러려면 **refresh token 확보**가 필요하기 때문이다. 두 방식 모두 `SocialClient` 전략 뒤에 캡슐화되어 코어 흐름은 불변.

**6. 공급자 간 동일인은 자동 병합하지 않는다.**
공급자마다 사용자 ID가 다르고 공유 식별자가 없다. 이메일은 (결정 4대로) 식별에 쓰지 않으므로 **이메일 기반 자동 병합도 하지 않는다**(오병합 위험). 서로 다른 공급자 로그인은 **별개 회원**이며, 동일인으로 묶으려면 로그인 상태에서의 **명시적 계정 연결**만 허용한다(후순위, 1:N 스키마로 지원).

**7. 공급자 프로필은 소셜 계정에 스냅샷으로 저장하고 서비스 신원과 분리한다.**
카카오 닉네임·프로필 이미지 등 공급자 프로필은 `social_account`에 저장하고, **서비스 닉네임(`member.nickname`, 온보딩에서 설정)과 분리**한다. 공급자마다 프로필이 다르고 member는 서비스 고유 신원이어야 하기 때문이다.

**8. 탈퇴는 공급자 해제를 선행한다.**
탈퇴 시 **공급자 revoke(애플)·unlink(카카오)를 먼저** 호출하고, 성공했을 때만 내부 데이터를 정리한다. 공급자에 연결이 남는 사고를 방지한다.

## 검토한 대안

### 대안 1 — 회원 테이블에 공급자 정보 직접 보관 (단일 테이블)
단순하지만 **한 회원의 다중 공급자 연결이 불가능**하다. 연결 요구로 회원↔소셜 계정 분리를 택함.

### 대안 2 — 서버 주도 OAuth2 리다이렉트 로그인
서버가 브라우저 리다이렉트로 공급자 인증을 주도하는 방식은 웹 전제라 **네이티브 앱에 부적합**하다. 앱이 공급자 SDK로 로그인하고 **서버는 전달받은 토큰을 검증**하는 방식을 택함.

### 대안 3 — 공급자 토큰을 서비스 인증에 계속 사용
매 API에 공급자 토큰을 쓰면 **서비스가 공급자 정책·수명에 종속**되고, 다중 공급자·자체 세션 통제가 어렵다. 식별 후 자체 토큰 발급([ADR 0009](0009-authentication-authorization.md))으로 분리.
