# ADR 0009: 사용자 인증/인가 전략

## 상태 (Status)

Accepted (2026-07-03)

## 배경

소셜 로그인 기반 모바일 서비스다. 클라이언트는 소셜 공급자(카카오·애플 등) 토큰을 서버에 전달해 로그인하고, 이후 서비스 API를 호출한다. 서버는 다음을 결정해야 한다.

- 로그인 이후의 서비스 API 요청을 **무엇으로 인증**할 것인가
- 로그아웃·탈퇴·강제 만료 시 **이미 발급된 세션을 어떻게 끊을** 것인가
- 리소스 접근 권한(마이페이지·운전기록·그룹 등)을 **어디서 판단**할 것인가

신원 확인(공급자 토큰 검증) 방법 자체는 [ADR 0008](0008-social-login.md)에서 다룬다. 본 문서는 **서버가 사용자를 어떻게 인증 상태로 유지하고 인가를 어디서 판단하는가**에 대한 결정이다.

## 결정

**1. 서버 자체 토큰 발급 — 소셜 공급자 토큰은 로그인 시 회원 식별에만 사용한다.**
공급자 토큰은 최초 로그인에서 내부 회원을 식별하는 용도로만 쓰고, 이후 모든 서비스 API 인증에는 **서버가 발급한 자체 토큰**을 사용한다. 서비스 인증이 공급자의 정책·토큰 수명에 종속되지 않게 하기 위함이다.

**2. access는 stateless, refresh는 서버가 상태를 추적한다.**
- **access token**: 짧은 만료의 stateless JWT. 서버 조회 없이 검증하여 평상시 요청 부담을 최소화한다.
- **refresh token**: 서버가 상태를 저장·추적한다. **재발급·로그아웃·탈퇴 시 서버가 무효화**할 수 있어야 한다는 요구를 충족한다.
- 두 토큰의 역할을 분리해 평상시 요청은 무상태로 빠르게, 세션 종료는 서버가 통제한다.

**3. refresh 상태는 현재 인프라(DB) 안에서 관리한다.**
별도 저장소(Redis 등) 대신 기존 DB에 둔다. 조회 비용을 감수하는 대신 **새 인프라·장애 지점을 늘리지 않고** 무효화·추적을 확보한다. (부하 증가 시 재검토 — 아래 대안)

**4. 인가는 서비스 계층에서 판단하고, 회원 식별자는 커스텀 애노테이션으로 주입한다.**
컨트롤러는 인증된 **회원 식별자만 주입**받는다. 이 주입은 **커스텀 애노테이션(`@CurrentUser`)** 으로 표준화해, 컨트롤러가 토큰 파싱·시큐리티 컨텍스트 접근 같은 보안 배관 코드 없이 **인증된 회원 id만** 받도록 한다. 실제 접근 권한(리소스 소유·참여 관계)은 **서비스 계층에서 회원 id와 리소스 관계를 대조**해 검증한다. 권한 판단을 도메인 로직 가까이 두어 규칙의 일관성·재사용성을 높인다. (리졸버 구현 방식은 구현 단계에서 정한다)

> 필터·시큐리티 설정, 토큰 만료값, 서명/해시 방식, 저장 스키마 등 구현 상세는 본 ADR 범위 밖이며 구현 단계에서 정한다.

## 결과

### 긍정적
- 평상시 API는 무상태 검증이라 수평 확장에 유리하다.
- 로그아웃·탈퇴·강제 만료 시 refresh를 무효화해 **세션을 서버가 실제로 끊을 수 있다.**
- 서비스 인증이 소셜 공급자 토큰 수명·정책에 비종속이다.
- 인가가 서비스 계층에 모여 리소스 권한 규칙이 일관된다.

### 부정적
- refresh 상태 관리(저장·무효화·정리)에 비용이 든다.
- access는 만료 전 개별 무효화가 어렵다 → 짧은 만료로 노출 창을 줄이는 선택에 의존한다.
- refresh 검증 시 저장소 조회 비용이 발생한다.

### 중립적
- refresh 저장소는 추후 Redis 등으로 전환할 여지를 남긴다.
- 만료·저장 방식 등 구체 파라미터는 운영 중 조정 가능한 결정으로 남긴다.

## 검토한 대안

### 대안 1 — access token 단일 사용 (refresh 없음)
무상태라 단순하지만 **로그아웃·강제 만료 대응이 어렵다**(발급 후 만료까지 서버 통제 불가). 세션 종료 통제 요구로 미채택.

### 대안 2 — refresh를 Redis에 저장
TTL 만료 관리가 편하다. 그러나 초기 단계에서 **새 인프라와 장애 지점**을 늘린다. 현재 인프라(DB)로 무효화·추적이 가능하므로 미채택하고, 부하 증가 시 재검토한다.

### 대안 3 — 인가를 컨트롤러/필터 단에서 판단
진입점에서 일괄 차단해 단순해 보이나, 소유·참여 같은 **도메인 관계 기반 권한**은 데이터 맥락이 필요해 서비스 계층이 적합하다. 컨트롤러단 검증은 규칙이 흩어지고 재사용이 어렵다.
