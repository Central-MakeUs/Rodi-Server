# 기능 스펙 작성 (spec-writer)

Rodi의 기능 스펙을 `docs/specs/`에 작성·보강하는 절차다. 구현 전 설계를 문서로 확정하는 단계이며, **코드는 작성하지 않는다.** 사용자와 주고받으며 스펙을 완성한다.

## 절차

1. **맥락 파악**: `docs/specs/`의 기존 스펙, 관련 도메인 코드, `CLAUDE.md`를 읽어 컨벤션·현재 구조를 파악한다.
2. **양식 복사**: `docs/specs/TEMPLATE.md` 구조로 `docs/specs/<기능-슬러그>.md`를 작성한다 (예: `member-auth.md`, `place-search.md`).
3. **인덱스 갱신**: `docs/specs/README.md` 목록 표에 한 줄(기능 | 설명 | Status) 추가.
4. **리뷰 대기**: 작성 후 사용자 리뷰·승인을 기다린다. 승인 없이 구현 단계로 넘어가지 않는다.

## 작성 원칙

- **완료 조건은 검증 가능하게.** "잘 동작한다"가 아니라 "현 위치 반경 1km 내 장소를 거리순으로 반환한다"처럼 테스트로 확인 가능한 문장.
- **불명확하면 추측 금지.** 모호한 부분은 임의 결정하지 말고 "미해결 질문" 섹션에 정리해 사용자에게 확인을 요청한다. (CLAUDE.md: 확실한 설계 이전에 코드를 짜지 않는다)
- **API 명세**: Method·Path·인증 여부·요청/응답 예시(JSON). 인증은 OAuth2→JWT 흐름.
- **도메인 모델**: 엔티티/VO와 핵심 필드·관계. 공간 데이터는 **SRID 4326(WGS84)** 명시.
- **Status 이력**: Status 표에 날짜·상태·변경 내용을 한 줄로 남긴다. 최초 작성은 `Draft`.

## 하지 않는 것

- 실제 코드 작성·수정(controller/service/entity 등) — 구현 단계의 일이다.
- 스펙 승인 전 다음 단계 진행.
